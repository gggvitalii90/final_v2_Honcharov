## Этап 1 — Очистка данных

### 🔴 Deals: аномальный SLA (время ответа менеджера)
**Находка:** SLA max = 26 908 464 сек ≈ **311 дней**. Кто-то ждал ответа менеджера почти год. Среднее SLA сильно искажено такими выбросами — при анализе использовать медиану.

**Решение:** SLA приведён к секундам (int). Значения не трогаем — выбросы анализируем отдельно.

**На слайде:** гистограмма SLA с длинным хвостом — наглядный пример почему mean ≠ типичное значение.

### 🔴 Deals: дата закрытия раньше даты создания

**Решение:** Значения не меняем. Добавлен флаг `closing_date_error`. При анализе временных рядов эти строки фильтруем отдельно.

**На слайде:** пример того, что данным из CRM нельзя доверять слепо — нужна проверка.

**Решение:** Значения не меняем. Добавлен флаг `amount_error`. При расчёте выручки и среднего чека — фильтруем.

**На слайде:** честно обозначить ограничение — это показывает зрелость аналитика.

## Этап 2 — Описательная статистика

### Ключевые находки для слайдов

**IAP — IQR = 0 (нестандартная ситуация):**
Более 50% первых платежей — ровно 1 000 €. Q1 = Q3 = 1 000 €, IQR = 0. Метод IQR в таком случае неинформативен — всё что не равно 1 000 € считается "выбросом". Использовали 5-й и 95-й перцентили вместо IQR.

**OTA — выбросов нет:**
Q1 = 5 000 €, Q3 = 11 000 €, IQR = 6 000 €. Нижняя граница = −4 000 € (ниже нуля), верхняя = 20 000 € (выше реального макс 11 500 €). На боксплоте — усы до min/max, точек-выбросов нет. Широкий IQR = разные продукты с разными ценами.

**Course duration — IQR = 0, данные биmodальные:**
11 мес (стандартный курс) + 6 мес (короткий курс) — две разные продуктовые линейки. 6 месяцев — не выброс, а отдельный продукт.

**SLA — обрезка оси до 72 ч:**
Макс SLA = 7 474 ч (311 дней). Один такой выброс делает гистограмму нечитаемой. Показываем на графике только 94.5% лидов (≤ 72 ч), остальные анализируем отдельно.

**Webinar — лучший канал по эффективности:**
Конверсия 6.86% — самая высокая среди всех источников. Расходы всего 2 874 € за год. Google Ads потратил 57 799 € при более низкой конверсии. Вывод для юнит-экономики: Webinar даёт самый дешёвый привлечённый клиент (CAC).

**Системная проблема дозвона:**
`Doesn't Answer` — причина №1 потери сделок (4 074 случая, 26% всех Lost). Одновременно 16.7% исходящих звонков не получают ответа (`Unattended Dialled`). Это не случайность — лиды не берут трубку массово. Решение: быстрый первый звонок (SLA ≤ 1 ч) и многоканальный контакт.

**71.8% лидов нецелевые (D+E):**
Маркетинг генерирует объём, а не качество. Лиды категорий A и B конвертируются кратно лучше, но их доля мала. Улучшение таргетинга без увеличения бюджета — главный рычаг роста конверсии.

## Глоссарий кода — объяснения для защиты

Здесь всё что может спросить преподаватель по коду:

| Конструкция | Что значит |
|---|---|
| `iap.dropna()` | Убирает строки где значение пустое (NaN) |
| `iap[~iap.isin([0, 1, 9])]` | Оставляет всё кроме значений 0, 1, 9. `~` = NOT, `isin` = входит в список |
| `iap[(iap != 0) & (iap != 1) & (iap != 9)]` | То же самое через обычные условия. `&` = И |
| `value_counts(normalize=True) * 100` | Частоты как доли (0–1), умноженные на 100 = проценты |
| `reindex(quality_order)` | Переставляет строки в нужном порядке (A→B→C→D→E), а не по частоте |
| `fillna(0)` | Заменяет NaN на 0 — нужно когда в числителе 0 сделок и pandas ставит NaN вместо 0% |
| `[::-1]` | Разворачивает список задом наперёд. Нужен потому что `barh` рисует первый элемент снизу, последний сверху — то есть перевёрнуто. Данные уже отсортированы по убыванию, `[::-1]` разворачивает их перед передачей в график чтобы наибольшее значение оказалось сверху |
| `.mul(100)` | Умножение — то же что `* 100`. Заменено на `* 100` для читаемости |
| `plt.savefig('path/file.png', dpi=100)` | Сохраняет график в файл. `dpi=100` — качество (точек на дюйм) |
| `deals.loc[условие, 'столбец']` | Выбирает строки по условию И одновременно конкретный столбец |
| `groupby('Col', observed=True)` | `observed=True` — без этого pandas выдаёт FutureWarning. Параметр говорит показывать только реальные категории, а не все возможные |
| `groupby().size()` | Считает количество строк в каждой группе |
| `paid / total * 100` | Конверсия: сколько оплатили из всех лидов, в процентах |
| `sort_values(ascending=False)` | Сортировка по убыванию (от большего к меньшему) |
| `q3 + 1.5 * iqr` | Верхняя граница выбросов по методу IQR. Всё выше — статистический выброс |
| `q1 - 1.5 * iqr` | Нижняя граница выбросов по методу IQR. Всё ниже — выброс |
| `dropna=False` в value_counts | Считать NaN как отдельную категорию, не игнорировать |
| `ax.tick_params(axis='x', rotation=90)` | Поворот подписей по оси X на 90° — подпись стоит строго под своим баром. При 45° текст визуально "съезжает" вправо и кажется что подпись не под баром |
| `fig, ax = plt.subplots(figsize=(10, 4))` | Создаёт одно полотно и один график. Используем даже для одного графика потому что `ax` даёт объектный контроль: `ax.set_title()`, `ax.bar_label()`, `ax.tick_params()`. `plt.subplots(1, 2)` — два графика рядом |
| `ax.bar_label(bars, fmt='%.0f', padding=4)` | Подписывает значения прямо на столбцах графика. `bars` — объект который вернул `ax.bar()` или `ax.barh()`. `fmt='%.0f'` — формат числа (целое без дробной части). `padding=4` — отступ от края столбца |
| `CTR * 100` | CTR (Click-Through Rate) = Clicks / Impressions. Умножаем на 100 чтобы получить проценты. Например 0.012 → 1.2% |
| `.to_string(index=False)` | Печатает таблицу (DataFrame) как текст в вывод ячейки **без столбца индекса** (без номеров строк слева). Просто для аккуратного вида распечатки, на расчёты не влияет |
| `.apply(lambda x: pd.Series({...}))` | Для каждой группы `groupby` считает СРАЗУ несколько метрик и складывает их в колонки. Например по каждому каналу: число лидов, оплат, конверсию — за один проход |
| `pd.merge(a, b, on='Source', how='left')` | Соединяет две таблицы по общему столбцу (как JOIN в SQL). `how='left'` — берём все строки левой таблицы; `inner` — только совпавшие |
| `np.where(условие, A, B)` | Векторное «если»: где условие истинно → A, иначе → B. Используем для CAC: если оплат > 0 → Spend/оплат, иначе пусто (NaN) |
| `nlargest(3, 'CAC')` / `nsmallest(3, ...)` | Топ‑3 / антитоп‑3 строк по значению столбца — без ручной сортировки |
| `ax2 = ax1.twinx()` | Создаёт вторую ось Y справа на том же графике. Нужно когда совмещаем две метрики с разными шкалами (например объём + конверсия %) |
| `.dt.to_period('M').dt.to_timestamp()` | Сводит дату к МЕСЯЦУ: `2024-04-17` → `2024-04-01`. Так группируем события по месяцам для временных рядов |
| `dtype={'Id': str, 'Contact Name': str}` в `read_csv` | Читает ID‑колонки как **строки**. Без этого 19‑значные ID превращаются в float и теряют точность (`5.8e+18`), ломая связи между таблицами и подсчёт клиентов |
| `DIVIDE(a, b)` (DAX, Power BI) | Деление, которое не падает при делении на 0 (возвращает пусто). Аналог `np.where`/`IFERROR`. Используем в мерах CAC, ROAS, C1 |

## Этап 3 — Анализ

**Временные ряды только за 1 год (июль 2023 – июнь 2024):**
Можем показать тренд по месяцам, но НЕ сезонность и НЕ сравнение год‑к‑году (`LY`/`YoY`) — данных за второй год нет. На защите честно проговорить это ограничение.

**Level of Deutsch — поле исключено из анализа:**
Свободный текст менеджера, не стандартизирован (`B1`, `б1`, `В1`, `beginner` — одно и то же). ТЗ (п. 3.7.2) просит анализ по немецкому, но корректно сделать его без нормализации в CRM нельзя. Решение и обоснование — в разделе 9.2 файла 3.

**Вечерний формат обучения конвертирует вдвое лучше утреннего:**
Evening — конверсия 42,3% при чеке 350 €, Morning — 22,9% при чеке 1 000 €. Гипотеза: вечерние группы — работающие, более мотивированные. Точка роста — тестировать расширение вечернего формата.

**Баг точности ID (ценная находка для защиты):**
При чтении очищенных CSV колонки `Contact Name`/`CONTACTID` читались как float и теряли точность → связь Deals/Calls с Contacts давала 0% совпадений. Исправлено через `dtype=str`. После фикса связь — 100%. Показывает глубину понимания работы с данными.

## Этап 4 — Юнит-экономика

**CPL ≠ CAC (разные метрики воронки):**
CPL (цена лида) = Spend / Лидов — верх воронки. CAC (цена клиента) = Spend / Оплат — низ воронки. Google Ads: дорогой лид (14,68 €) И низкая конверсия → высокий CAC (334 €). Webinar: средний CPL (7,62 €), но лучшая конверсия → низкий CAC (111 €).

**Webinar — точка масштабирования:**
ROAS 19,19x, CAC 111 € при минимальных расходах (2 874 €). Канал не исчерпан (377 лидов). Главная рекомендация — увеличить бюджет на вебинары.

**Дерево метрик считается в Google Sheets формулами:**
В Python агрегируем сырьё (UA, B, T, Revenue, AC по каналам/продуктам), а сам метрик‑три (C1, AOV, ARPPU, APC, CLTV, LTV, CAC, ROAS) считается формулами в таблице — как на уроках продактаналитики. Power BI читает готовый результат.
