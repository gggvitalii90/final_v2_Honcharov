# Project Decisions

Короткая фиксация спорных вопросов и решений, которые были отдельно обсуждены по проекту. Этот файл нужен, чтобы не терять контекст между ноутбуками, Google Sheets, Power BI и презентацией.

## 1. Где считать unit economics

**Вопрос:** считать unit economics в Python или в Google Sheets?

**Решение:** финальная unit economics считается в Google Sheets. Python отвечает за очистку и подготовку данных, а не за финальные продуктовые расчёты. В Google Sheets загружаем очищенные `deals_clean` и `spend_clean`, после чего там считаем метрики, дерево метрик, точки роста и гипотезы.

## 2. Какие данные грузить в Google Sheets

**Вопрос:** грузить все таблицы или только нужные для unit economics?

**Решение:** для базовой unit economics нужны только очищенные Deals и Spend. Contacts и Calls не грузим в Google Sheets для базового расчёта CAC / ROAS / C1 / среднего платежа. Calls и Contacts остаются для анализа звонков, менеджеров и связей в Python.

## 3. Роль `04_unit_economics.ipynb`

**Вопрос:** что теперь делает четвёртый ноутбук, если расчёты уходят в Google Sheets?

**Решение:** `04_unit_economics.ipynb` подготавливает и валидирует clean-датасеты для Google Sheets. В Sheets загружаются полноценные очищенные `deals_clean.csv` и `spend_clean.csv`; мы ограничиваем только количество таблиц, а не режем колонки. Финальные формулы unit economics считаются в самой Google Таблице.

## 4. Структура Google Sheets

**Вопрос:** как организовать таблицу, чтобы не было хаоса?

**Решение:** raw-листы в Sheets не редактируются руками. Рекомендуемая структура: `01_deals_clean_raw`, `02_spend_clean_raw`, `03_methodology`, `04_unit_by_source`, `05_unit_by_campaign`, `06_unit_by_product`, `07_monthly_unit`, `08_metrics_tree`, `09_hypotheses`, `10_dashboard_export`.

## 5. Правило оплаченной сделки

**Вопрос:** что считать оплаченной сделкой для финансового анализа?

**Решение:** оплаченная сделка для расчётов = `Stage = Payment Done` и `Initial Amount Paid` не пустой. Если есть только статус без суммы или сумма без корректного статуса, такая строка не участвует в финансовых расчётах как оплаченная.

## 6. Revenue vs First Payment

**Вопрос:** можно ли называть `Initial Amount Paid` выручкой?

**Решение:** в базовой аналитике не называем `Initial Amount Paid` полной выручкой. Это `First Payment Amount` / "сумма первого платежа". Полную или признанную выручку считаем отдельно в сценарном блоке revenue recognition.

## 7. Revenue recognition

**Вопрос:** как учитывать рассрочку, будущие платежи и возможные возвраты?

**Решение:** это отдельный сценарный анализ, а не замена базовой метрики. Для `Recurring Payments` можно моделировать будущие платежи как `(Offer Total Amount - Initial Amount Paid) / (Course duration - 1)`. Для `One Payment` можно отдельно оценить recognized revenue и refund risk, если клиент учился меньше полной длительности курса. Возвраты в данных не видны, поэтому говорим только о модели риска, а не о фактических возвратах.

## 8. Power BI и источник данных

**Вопрос:** откуда Power BI должен брать данные?

**Решение:** для итоговых продуктовых метрик Power BI лучше подключать к `10_dashboard_export` из Google Sheets или к аккуратному экспорту из него. Тогда Power BI визуализирует согласованные метрики, а не пересчитывает логику заново.

## 9. Parquet для Power BI

**Вопрос:** нужно ли дополнительно сохранять очищенные данные в Parquet?

**Решение:** да, для Power BI создаём дополнительные clean-файлы: `deals_clean.parquet`, `spend_clean.parquet`, `calls_clean.parquet`, `contacts_clean.parquet`. CSV остаётся для проверки, Git и Google Sheets; Parquet нужен для типов данных, скорости и удобства Power BI.

## 10. Power BI MCP

**Вопрос:** сможет ли Codex помогать с Power BI через MCP?

**Решение:** Power BI MCP может помочь с моделью, DAX, проверками и подключением данных, но финальная визуальная раскладка report canvas может потребовать ручной настройки в Power BI Desktop. Полностью автоматическую сборку красивого `.pbix` с нуля не считаем надёжной основной стратегией.

## 11. Docker

**Вопрос:** стоит ли делать проект "одной кнопкой" через Docker?

**Решение:** Docker полезен для автоматизации data pipeline: очистка, экспорт CSV/Parquet, обновление данных. Но строить полноценный визуальный Power BI Desktop-отчёт из Docker для этого проекта избыточно и рискованно. Основная стратегия: Docker/Python автоматизируют данные, Power BI один раз настраивается и дальше обновляется.

## 12. Менеджеры: first-touch vs closer

**Вопрос:** корректно ли сравнивать менеджеров одним рейтингом, если клиенту могли звонить разные люди?

**Решение:** перед финальным рейтингом менеджеров надо добавить анализ ролей: first-touch manager, участник коммуникации, deal owner / closer, payment closer. Если менеджеры выполняют разные роли, рейтинг нужно разделить по группам или интерпретировать общий рейтинг очень осторожно.

## 13. Quality и менеджеры

**Вопрос:** можно ли напрямую оценивать менеджера по Quality его лидов?

**Решение:** нельзя делать жёсткий вывод "у менеджера плохая конверсия, потому что у него плохие лиды". `Quality` субъективен, а с одним клиентом могли работать разные менеджеры. Используем Quality как дополнительный срез входящего потока, а не как прямую оценку эффективности конкретного менеджера.

## 14. Сколько звонков до оплаты

**Вопрос:** сколько звонков сделали клиенту до того, как он стал клиентом?

**Решение:** это отдельный анализ в `03_analysis.ipynb`, а не просто описательная статистика. Для оплаченных клиентов (`Payment Done` + сумма первого платежа) считаем количество звонков по `CONTACTID` до `Closing Date`. Если `Closing Date` пустой, такие строки нужно отдельно обработать или исключить из анализа времени до оплаты.

## 15. Навигация по проекту

**Вопрос:** нужна ли навигация внутри ноутбуков и отдельный файл навигации?

**Решение:** делаем оба уровня. Внутри каждого `.ipynb` нужен кликабельный план, ручные якоря и ссылка "вернуться к плану". Отдельно создаётся `PROJECT_NAVIGATION.md` с картой: пункт задания -> файл проекта -> что смотреть. Внешние ссылки из `.md` прямо на якорь внутри `.ipynb` могут работать нестабильно, поэтому основная надёжная навигация должна быть внутри самих ноутбуков.
