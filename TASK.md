# Задание финального проекта — Data Analytics (IT Career Hub)

> Составлено из PDF: `Python for DA_Pr_Project (6).pdf` (онбординг), `Python for DA_LfS_Project.pdf` (описание проекта + FAQ + критерии), `Python DA_Org Project (2).pdf` (организационный регламент).

## 1. Концепция проекта

Вы — аналитик данных в крупной компании и получили новый проект от клиента — **онлайн-школы программирования X**. Проект включает работу с данными CRM-системы, которая ведёт учёт статусов лидов и сделок.

**Цель проекта:** очистка и анализ данных из CRM-системы для повышения эффективности работы онлайн-школы программирования.

## 2. Порядок работы (важно!)

> Сначала выполняется задание по **Python DA** — это даст общее понимание бизнеса и его текущей ситуации. Затем выполняется задание по **продуктовой аналитике** с учётом текущей ситуации бизнеса.

**Уникальность проекта:** обязательны основные требования и условия, но выбор инструментов (из пройденных на курсе) — за студентом:
- Всесторонний отчёт: Google Docs / Google Sheets / изученные инструменты визуализации.
- Дашборд: Python DA / Tableau / Power BI.

## 3. Задание по Python DA

### 3.1 Очистка и подготовка данных
1. Удалить дубликаты и неактуальные столбцы.
2. Соответствующим образом обработать отсутствующие значения.
3. Преобразовать типы данных (даты, числовые значения).

### 3.2 Описательная статистика
1. Рассчитать сводную статистику (среднее, медиана, мода, диапазон) для числовых полей.
2. Проанализировать категориальные поля: качество (Quality), стадия (Stage), источник (Source), продукт (Product).

### 3.3 Анализ временных рядов
1. Проанализировать тенденцию создания сделок с течением времени и их связь со звонками.
2. Изучить распределение времени закрытия сделок и продолжительность периода от создания до закрытия.

### 3.4 Анализ эффективности кампаний
1. Сравнить эффективность различных кампаний: генерация лидов и коэффициент конверсии.
2. Оценить эффективность маркетинговых источников (Source) в генерации качественных лидов.

### 3.5 Анализ эффективности работы отдела продаж
1. Оценить эффективность отдельных владельцев сделок и рекламных кампаний: количество обработанных сделок, коэффициент конверсии, общая сумма продаж.

### 3.6 Анализ платежей и продуктов
1. Изучить распределение типов оплаты и их влияние на успешность сделок.
2. Проанализировать популярность и успешность различных продуктов и типов обучения.

### 3.7 Географический анализ (✱ дополнительный блок)
1. Проанализировать географическое распределение сделок по городам.
2. Изучить влияние уровня знания немецкого языка на успешность сделок в разных городах.

### Результаты части Python DA
1. Очищенный набор данных.
2. Всесторонний отчёт с подробным описанием процесса анализа, выводов, визуализации и рекомендаций.
3. Дашборд для изучения данных и выводов.
4. Презентация с кратким изложением основных выводов и рекомендаций.

## 4. Задание по продуктовой аналитике

**Задача:** найти точку роста бизнеса, сформулировать гипотезу улучшения бизнес-процесса для роста метрик и описать механику тестирования (тест не должен занимать больше 2-х недель).

1. Посчитать юнит-экономику по продуктам.
2. Из юнит-экономики определить точки роста бизнеса.
3. Понять дерево метрик для бизнеса.
4. Понять, на какую метрику продукта будет воздействие, и сформировать гипотезы.
5. Описать метод проверки гипотез с формулированием условий проведения.

## 6. FAQ — правила интерпретации данных (критично для очистки!)

| Вопрос | Ответ преподавателя/заказчика |
|---|---|
| Что значит дата в названии Campaign (напр. `02.07.23wide_DE`)? | Campaign — внутреннее название, придумывают маркетологи, правил формирования нет. На даты в названии не ориентироваться. Лиды связаны с этим столбцом — он показывает, с какой кампании пришёл лид. |
| Из чего складывается SLA? | Время ответа продажника на заявку: сколько прошло с момента подачи заявки до контакта. |
| Stage = Payment Done? | Деньги получены на счёт. Остальные статусы для анализа не важны — их можно перевести и интерпретировать. |
| Initial Amount Paid — это что? | Сколько клиент заплатил первым платежом (вся сумма или часть). Если Payment Done — считаем, что эта сумма была заплачена. Ошибки в поле аналитик исправляет сам и обосновывает решение. |
| Initial Amount есть, а статус Lost? | Да, бывает: факт сделки определяем по совокупности столбцов. |
| Статус оплачен, а сумма пустая? | Оплата была, просто сумма не зафиксирована. |
| Значения 0, 1, 9 в Initial/Total Amount? | Демо-доступы за символическую плату. |
| Lost Reason = Duplicate? | Лидов было два, в одном стоит Lost, потому что это дубликат. |
| Статус «ожидание платежа» и т.п.? | Оплата ещё не прошла. Факт оплаты понимаем из статуса. Столбец суммы — «предложение на первую оплату». |
| Есть платежи, но человек не обучается (нет Months of study, нет Payment Done)? | Если месяц не заполнен — студент не учится; деньги его учитывать нельзя (неважно: ошибка в данных или возврат). |
| Initial Amount > Total Amount? | Очевидная ошибка. Аналитик принимает решение о способе исправления. |
| Months of study — продолжает ли учиться? | Это сколько студент учится к моменту создания датасета; продолжает или нет — определить нельзя. |
| Closing Date — что это? | Дата закрытия = дате поступления денег = Payment Done. Сделка закрыта, когда менеджер прекратил общение с клиентом; для студентов — дата оплаты. Если дата закрытия раньше открытия — ошибка, аналитик решает и обосновывает исправление. |
| Поле Quality — объективное? | Заполняется субъективно менеджером по продажам. |
| Payment Type / Initial Amount Paid могут содержать ошибки? | Да, заполняются вручную; определить, в чём именно ошибка при расхождении, невозможно. |
| Критерий «успешной сделки»? | На поверхности — Payment Done. Но если у аналитика есть свои идеи определения успешности — он может и должен обосновать, просчитать и включить в рассказ. |
| Что такое deal? | Корректнее «потенциальная сделка» — речь о работе менеджеров с клиентом. |
| Какие столбцы заполнялись вручную/автоматически? | Выводы делаются из изучения данных — это результат вашей работы. |

## 7. Регламент и сроки

- **Виды занятий:** онбординг → консультации → самостоятельная доработка → защита.
- **Загрузка:** проект загружается в LMS (модуль Project) **за 1 день до презентации** — ссылка на проект в соответствующее задание.
- **Защита:** презентация + ответы на вопросы преподавателя, **5–10 минут**, свободная форма. Главное — продемонстрировать самостоятельность выполнения.
- **Рекомендованная структура презентации:** приветствие → описание проекта/задания → описание результата → основные сложности/открытия.
- ДЗ и итоговые тесты по модулям сдаются до воскресенья недели защиты. Финальный тест по курсу — до воскресенья после защиты.

## 8. Критерии оценки

7 критериев, каждый оценивается от 1 (отлично) до 5 (неудовлетворительно). Шесть вносят равный вклад, «Вопросы преподавателя» учитываются **трижды**:

**Оценка за проект = сумма оценок по каждому критерию / 9** (округляется до 0.5 включительно в меньшую сторону).

| # | Критерий | На «1» (отлично) |
|---|---|---|
| 1 | Очистка и подготовка данных | Удалены дубликаты, обработаны пропуски, типы приведены корректно |
| 2 | Описательная статистика | Все показатели (mean, median, mode, range), анализ категориальных, визуализации и выводы |
| 3 | Анализ данных | Временные ряды, продажи, источники, продукты; визуализировано; обоснованные выводы |
| 4 | Юнит-экономика и дерево метрик | Рассчитана по продуктам, построено дерево метрик, выявлены точки роста |
| 5 | Формулирование и проверка гипотез | Обоснованная гипотеза, метод тестирования, метрика и условия эксперимента |
| 6 | Представление результатов | Наглядно, в едином стиле, корректно; выводы понятны без докладчика |
| 7 | Вопросы преподавателя (×3) | Ответ логичный и аргументированный, конкретика, корректные термины |

**Шкала итоговой оценки:** 1 — отлично (81–100%), 2 — хорошо (61–80%), 3 — удовлетворительно (41–60%), 4 — неуд. (21–40%), 5 — неуд. (до 20%).

**Важно:**
- Качественная реализация дополнительных функций может компенсировать незначительные недочёты основного функционала (до 0.1 балла на усмотрение преподавателя).
- При публикации проекта в портфолио **обязательно удалить ключи доступа к БД** — публикация ключей ухудшает оценку на 1 балл.
